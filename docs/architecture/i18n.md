## Locales

|                      |                                                    |
| -------------------- | -------------------------------------------------- |
| **Default locale**   | `es-AR` (Spanish, Argentina) — the source of truth |
| **Secondary locale** | `en-US` (English, US)                              |
| **Library**          | [next-intl](https://next-intl.dev) v4              |

Locales are defined in **two** places that must agree:

- `src/i18n/config.ts` → `i18n = { defaultLocale: "es-AR", locales: ["es-AR", "en-US"] }`
- `src/i18n/routing.ts` → `defineRouting({ locales: ["es-AR", "en-US"], defaultLocale: "es-AR" })`

If you add or change a locale, update both (and add the matching message file). `Locale` is a derived type (`(typeof i18n)["locales"][number]`); use it instead of `string` where possible.

## The `src/i18n` modules

| File         | Responsibility                                                                                                                                                                                                                                                          |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `config.ts`  | The locale list + `defaultLocale`, the `Locale` type, `localesPath`, and **`buildLocaleAlternates(path)`** — the hreflang/canonical map builder used by `lib/metadata.ts`.                                                                                              |
| `routing.ts` | `defineRouting(...)` and the navigation wrappers `createNavigation(routing)` exports (`Link`, `redirect`, `usePathname`, `useRouter`, `getPathname`). **Import `Link`/`redirect` from here**, not from `next/link` / `next/navigation`, so locale prefixing is handled. |
| `request.ts` | `getRequestConfig` — resolves the request locale (falling back to `defaultLocale` when missing/invalid) and lazy-imports the matching `public/locales/<locale>.json` as `messages`.                                                                                     |

## Message files (UI strings)

UI/chrome strings (button labels, form copy, metadata site-name, etc.) live in:

```
public/locales/es-AR.json
public/locales/en-US.json
```

Rules:

- **Every user-facing string exists in both files**, under the same key path. A key present in one and missing in the other is a bug — es-AR is the source, en-US is the translation.
- `request.ts` loads these per-request. In Server Components, read them with `getTranslations(namespace)` / `getMessages()` from `next-intl/server`; in Client Components use the `useTranslations` hook.
- Example consumer: `lib/metadata.ts` reads `getTranslations("Metadata")` and uses `t("site-name")` for the OpenGraph `siteName`.
- Keep these for **interface text only.** Long-form, editor-owned content (page bodies, blog posts, the Creed, mission text) is authored per-locale **in Contentful**, not here.

## Message-key const maps (`src/i18n/messageKeys/`)

Server code never returns localized prose. A route handler or Server Action returns a stable, locale-agnostic **`messageKey`**, and the client resolves it to copy via `useTranslations()`. Those keys are declared once, as typed const maps:

```
src/i18n/messageKeys/
  subscribeBanner.ts   → SUBSCRIBE_BANNER_KEYS  + SubscribeBannerKey   (+ .test.ts)
  contactForm.ts       → CONTACT_FORM_KEYS      + ContactFormKey       (+ .test.ts)
```

Rules:

- **Never hardcode a `messageKey` string literal at an emit site.** Import the const. A bare literal is not linked to the map, so a key rename desyncs the emitter silently and the visitor falls through to the generic error (this was ICR-148 — `/api/subscribe` was the last offender).
- Each map is a **zero-dependency leaf module** — a const + its derived union type, no React, no `"use client"`. That is what lets **any** tier import it: RSC, client component, Server Action, or route handler. It is also why the maps live here and not next to a component: a route handler must not have to reach into `src/components/*`.
- **No barrel `index.ts`.** Importers reference the leaf file directly. A barrel in `src/i18n/` invites re-exporting the server-only `request.ts` into client bundles.
- Each map has a **colocated test** pinning two things: the map's values are exactly the literal wire strings, and both locale JSONs carry the corresponding namespace keys. That test is the byte-level anchor of the wire contract — do not weaken it.
- The resulting contract is layered: the key-map test pins **map → wire string**, and each emitter's test pins **response → the right map entry**. Composed, the response is still pinned to the exact string, _and_ a key rename now fails to compile at the emitter. See `docs/architecture/forms-and-email.md` for the subscribe + contact flows that consume this.

## Middleware: `src/proxy.ts`

> ⚠️ The middleware file is named **`src/proxy.ts`** and exports a function named **`proxy`** — not the conventional `middleware`. Don't go looking for `middleware.ts`.

```ts
export function proxy(request: NextRequest) {
  if (request.method === "OPTIONS")
    return new NextResponse(null, { status: 200 }); // CORS preflight
  const ext = request.nextUrl.pathname.split(".").pop()?.toLowerCase();
  if (ext && safeExtensions.includes(ext)) return NextResponse.next(); // bypass static assets
  return intlMiddleware(request); // next-intl locale handling
}

export const config = { matcher: ["/((?!_next|_vercel|api|trpc).*)"] };
```

What it does, in order:

1. **Short-circuits CORS preflight** (`OPTIONS` → 200).
2. **Bypasses static assets** by extension (`ico, png, jpg, jpeg, gif, svg, webp, css, js, json, xml, txt, woff, woff2, ttf, eot`) so they don't get locale-prefixed.
3. **Delegates to `createMiddleware(routing)`** for everything else — this is what redirects `/` → `/es-AR`, validates the locale segment, and sets up the locale context.

The `matcher` excludes `_next`, `_vercel`, `api`, and `trpc` so internal/route-handler traffic skips the middleware entirely. API routes are **not** locale-prefixed.

## Locale alternates / hreflang (SEO)

`buildLocaleAlternates(path)` in `src/i18n/config.ts` returns `{ "es-AR": "<base>/es-AR/<path>", "en-US": "<base>/en-US/<path>" }` from `NEXT_PUBLIC_BASE_URL`. `lib/metadata.ts` plugs that into `alternates.languages` on every page (and sets `alternates.canonical` to the current locale's URL), so search engines and AI crawlers get correct hreflang signals. The OpenGraph `locale` is the same value with the dash swapped for an underscore (`es-AR` → `es_AR`). See [`seo-and-metadata.md`](./seo-and-metadata.md).

## Adding or changing a locale — checklist

1. Add the locale to **both** `src/i18n/config.ts` and `src/i18n/routing.ts`.
2. Create `public/locales/<new-locale>.json` with the full key set translated.
3. Author the corresponding per-locale content in Contentful (the getters pass `locale` through to GraphQL).
4. Confirm `buildLocaleAlternates()` now emits the new hreflang and that metadata looks right on a preview deploy.
5. Sanity-check `src/proxy.ts` redirects and that the new locale segment resolves.

## Gotchas

- **Don't import `Link`/`redirect`/`usePathname` from `next/*`** in localized UI — use the wrappers from `src/i18n/routing.ts` so locale prefixes are preserved.
- **A missing translation key** silently renders the key or falls back depending on next-intl config — keep both JSON files in lockstep; a Vitest smoke test over the two key sets is a cheap guard.
- **`request.ts` special-cases the default** (`locale === "es-AR" ? import("../../public/locales/es-AR.json") : import(\`…/${locale}.json\`)`) — when adding locales, ensure the dynamic import path still resolves your new file.
